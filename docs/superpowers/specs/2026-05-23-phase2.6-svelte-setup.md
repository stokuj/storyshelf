**Data:** 2026-05-23
**Zakres:** `frontend/` (usuwane), `svelte-frontend/` (nowe), `infra/`, `Makefile`, `.github/workflows/`, `CLAUDE.md`, `docs/ARCHITECTURE.md`
**Stan wyjściowy:** Backend rozszerzony (Phase 2.1–2.5 ukończone). Vue 3 SPA wciąż działa (`make dev-up` serwuje na `:5173`). CORS + cookies cross-origin gotowe od Phase 2.0 (`CORS_ALLOWED_ORIGINS`, `CORS_ALLOW_CREDENTIALS`, `JWT_COOKIE_SAMESITE`). OpenAPI snapshot wygenerowany w `docs/api/openapi.yml`. ## Kontekst

Phase 2 kończy się odcięciem Vue i wymianą stosu frontendowego na SvelteKit. Decyzja z grilling-sessions 2026-05-23: **Vue umiera w jednym commicie**, brak okresu przejściowego, brak dual-runningu (powód: hobby projekt, jeden użytkownik-developer, API rozszerzenia 2.1–2.5 nie zachowywały wstecznej kompatybilności z Vue klientem właśnie dlatego, że ten klient ma zniknąć przed implementacją widoków Svelte).

Phase 2.6 ma **dwie akcje** w jednym branchu (`phase/2.6-svelte-setup`):

1. **Akcja 1 — Vue removal:** wykonanie checklisty z ADR-004 (`docs/decisions/ADR-004-vue-removal.md`). Jeden commit, jedno żądanie pull. Po tym commicie repo nie ma śladów Vue.
2. **Akcja 2 — SvelteKit scaffolding:** nowy folder `svelte-frontend/` w monorepo z minimalnym działającym SvelteKit (stack zgodny z `svelte(wideframe)/handoff/CLAUDE.md`), serwowany przez Docker, połączony z API przez `hooks.server.ts` + typowany klient (`_client.ts`) + generowane typy OpenAPI.

**Phase 2.6 NIE implementuje widoków** (routes, komponenty UI, formularze). Kończy się "hello world" SvelteKit dostępnym pod `http://localhost:5174/`, który robi jeden uwierzytelniony fetch do `GET /api/users/me/` z passthrough cookies przez `hooks.server.ts` i wyświetla wynik. To dowodzi że infrastruktura, CORS, cookies i typy działają end-to-end. Faktyczna implementacja widoków to Phase 2.7–2.9.

**Decyzja: monorepo, folder `svelte-frontend/`.** Nie tworzymy osobnego repozytorium. Powody: (1) hobby projekt, jeden deweloper — overhead drugiego repo nie zwraca się; (2) CI ma jeden workflow, jeden release; (3) docs/specs/ADR żyją w jednym miejscu; (4) `svelte(wideframe)/handoff/` zakłada layout `src/` względem korzenia frontu — łatwo zmapować na `svelte-frontend/src/`.

    import type { Handle, HandleFetch } from '@sveltejs/kit';
    import { INTERNAL_API } from '$lib/config';

    export const handleFetch: HandleFetch = async ({ event, request, fetch }) => {
      // Re-route public API URL → internal Docker service for server-side fetch
      if (request.url.startsWith(event.url.origin + '/internal-api/')) {
        const target = request.url.replace(event.url.origin + '/internal-api', INTERNAL_API);
        request = new Request(target, request);

      return fetch(request);
    };

## Obszar 4 — Połączenie z API (smoke-test end-to-end)

### Stan oczekiwany po implementacji

1. `make dev-up` startuje wszystkie serwisy włącznie z `svelte`.
2. `http://localhost:5174/` ładuje stronę.
3. Bez sesji: strona pokazuje link do `/login` (route nie istnieje jeszcze — to OK, smoke-test nie wymaga).
4. Po zalogowaniu via curl (`POST http://localhost:8000/api/auth/login/` z cookie save → przekazanie cookie do `http://localhost:5174/` jako `Cookie:` header) — strona pokazuje `Hello, <username>`.

To dowodzi:
- SvelteKit działa w Dockerze.
- `hooks.server.ts` przekazuje cookies z requestu klienta do server-side fetch do API.
- CORS + `SameSite=Lax` z Phase 2.0 wystarczają dla server-side fetch (server-side nie podlega CORS, ale klient-side musi działać z cross-origin).
- `INTERNAL_API_URL` (kontener `django:8000`) działa z poziomu kontenera `svelte`.
- Generator `openapi-typescript` produkuje używalne typy.

### OpenAPI types — proces

1. `npm run types:api` czyta `../docs/api/openapi.yml` (zacommittowany od Phase 2.0).
2. Generuje `src/lib/types/api.generated.ts` (gitignored).
3. Plik eksportuje typ `paths`, `components['schemas']['User']`, itd.
4. `src/lib/api/_client.ts` ma signature `apiFetch<T>(...)` — `T` typowany przez generic, agent Phase 2.7+ pisze:
   ```ts
   import type { components } from '$lib/types/api.generated';
   type User = components['schemas']['User'];
   ```

### `hooks.server.ts` — model auth

Phase 2.0 ustaliło: JWT access w pamięci, refresh w HttpOnly cookie (`refresh_token`). **W kontekście SvelteKit SSR** — access token przestaje być "w pamięci" (nie ma persistent client) i przenosimy się na model **server-side session**: SvelteKit dostaje cookies z przeglądarki, w `hooks.server.ts` przekazuje je do API przez `handleFetch`, API waliduje JWT cookie (refresh używany do silent refresh tak jak w ADR-001).

**To wymaga doprecyzowania w planie 2.6** — czy SvelteKit utrzymuje access token serverside (locals + secret cookie do clientside), czy każde wywołanie API przekazuje refresh cookie i polega na backendzie żeby zwrócił nowy access cookie. Decyzja preferowana: **drugi wariant** (passthrough), bo nie wymaga zmian backendu i nie wymaga drugiego secretu.

Konsekwencja: backend ADR-001 jest **kompatybilny w 100%** z SvelteKit SSR, ale prosimy o doprecyzowanie w `docs/decisions/ADR-001-jwt-httponly-cookies.md` że passthrough cookies przez SvelteKit nie jest "drugą warstwą sesji" — jest tylko transportem.

### Non-goals

- Faktyczne strony login/signup — Phase 2.7.
- Refresh token rotation logic — używamy tego co backend już robi.
- CSRF token handling — JWT cookies + SameSite chroni, brak Django sessions.
