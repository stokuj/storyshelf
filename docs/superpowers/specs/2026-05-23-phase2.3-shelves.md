# Phase 2.3 — Multi-shelf collections

**Data:** 2026-05-23
**Zakres:** `backend-django/shelf/`, `backend-django/users/` (publiczny endpoint shelvey), `backend-django/config/urls.py`
**Stan wyjściowy:** Istnieje pojedynczy model `ShelfEntry` z 3 statusami (WANT_TO_READ / READING / READ), bez możliwości tworzenia własnych kolekcji. Frontend kontrakt (`svelte(wideframe)/handoff/03-api-contract.md:67-73`) oczekuje endpointów `/me/shelves`, `/u/:handle/shelves`. Phase 2.0 zapewnia OpenAPI snapshot + CORS, Phase 2.1 dodaje slug/paginację na książkach — ta faza je konsumuje, ale ich nie zmienia.

---

## Kontekst

Frontend (SvelteKit, Phase 2.7+) wprowadza pojęcie **półki/kolekcji** jako nazwanej listy książek tworzonej przez użytkownika ("Ulubione kryminały", "Do przeczytania latem", "Prezenty na święta"). Jest to **odrębna koncepcja** od istniejących statusów czytania `ShelfEntry` (want/reading/read) — te zostają nietknięte i służą do śledzenia stanu czytania per książka.

Dwa problemy do rozwiązania:

1. **Brak modelu kolekcji** — istniejący `ShelfEntry` (`shelf/models.py:7`) modeluje **status czytania** (jedna książka → jeden status per user), nie wielokrotne tematyczne kolekcje. Frontend potrzebuje wielu nazwanych list, do których ta sama książka może należeć równolegle.
2. **Brak publicznego endpointu profilu z kolekcjami** — `UserProfileView` (`users/views.py:127`) zwraca tylko podstawowe pola (`username`, `bio`, `avatarUrl`, `memberSince`). Frontend kontrakt wymaga `GET /u/:handle/shelves`, który zwraca **tylko publiczne** półki danego użytkownika z poszanowaniem `profile_public`.

Phase 2.3 wprowadza model `Shelf` + `ShelfMembership` jako nową domenę współistniejącą z `ShelfEntry`, oraz endpointy CRUD-lite zgodne z kontraktem.

## Obszar 1 — Modele `Shelf` i `ShelfMembership`

### Stan obecny

- `shelf/models.py:7` zawiera tylko `ShelfEntry` (status czytania, `unique_together(user, book)`)
- Brak modeli typu "nazwana kolekcja"
- `shelf/migrations/0002_*.py` — ostatnia migracja na constraintach `ShelfEntry`

### Planowane zmiany

1. **Nowy model `Shelf`** — `shelf/models.py`:
   ```python
   class Shelf(models.Model):
       user = models.ForeignKey(
           settings.AUTH_USER_MODEL,
           on_delete=models.CASCADE,
           related_name="shelves",
           db_index=True,
       )
       name = models.CharField(max_length=80)
       slug = models.SlugField(max_length=80)  # generowany z name w save()
       description = models.TextField(max_length=500, blank=True, default="")
       is_public = models.BooleanField(default=False)
       created_at = models.DateTimeField(auto_now_add=True)
       updated_at = models.DateTimeField(auto_now=True)

       class Meta:
           constraints = [
               models.UniqueConstraint(fields=["user", "slug"], name="unique_user_shelf_slug"),
           ]
           ordering = ("-created_at",)
   ```
   - Slug generowany w `save()` przez `django.utils.text.slugify(self.name)` — kolizje rozwiązywane sufiksem numerycznym (`-2`, `-3`) per user (helper `_unique_slug_for_user()`).
   - `unique_together` na `(user, slug)` — slugi unikatowe **per user**, nie globalnie (dwóch userów może mieć "ulubione").

2. **Nowy model `ShelfMembership`** — `shelf/models.py`:
   ```python
   class ShelfMembership(models.Model):
       shelf = models.ForeignKey(Shelf, on_delete=models.CASCADE, related_name="memberships", db_index=True)
       book = models.ForeignKey("books.Book", on_delete=models.CASCADE, related_name="shelf_memberships", db_index=True)
       position = models.PositiveIntegerField(default=0)  # do ręcznej kolejności (drag-and-drop w przyszłości)
       added_at = models.DateTimeField(auto_now_add=True)

       class Meta:
           constraints = [
               models.UniqueConstraint(fields=["shelf", "book"], name="unique_shelf_book_membership"),
           ]
           ordering = ("position", "-added_at")
           indexes = [
               models.Index(fields=["shelf", "position"], name="shelfmembership_pos_idx"),
           ]
   ```
   - Pozycja **opcjonalna w MVP** — domyślnie `0`, sortowanie sekundarne po `-added_at`. Endpoint reorder poza Phase 2.3.

3. **Migracja** — `shelf/migrations/0003_shelf_shelfmembership.py` (auto-wygenerowana przez `makemigrations`). Nie modyfikuje `ShelfEntry`.

### Non-goals (Obszar 1)

- Endpoint reorderowania (`PATCH /me/shelves/:id/reorder`) — pole `position` istnieje, ale UI nie używa go w MVP
- Współdzielenie półek między userami (M2M owner) — single-owner wystarczy
- Soft-delete półek — `on_delete=CASCADE` na `Shelf` usuwa wszystkie membershipy

## Obszar 2 — Endpoints `/me/shelves/*`

### Stan obecny

- `shelf/urls.py` ma tylko dwa pathy: `""` (ShelfListView) i `"<int:book_id>/"` (ShelfEntryView) — oba dotyczą **starych** statusów czytania
- `config/urls.py:10` rooteuje `path("api/shelf/", include("shelf.urls"))`
- Brak endpointów typu "lista kolekcji usera"

### Planowane zmiany

1. **Rozszerzenie `shelf/urls.py`** — dodajemy podścieżki **bez konfliktu** z istniejącymi:
   ```python
   urlpatterns = [
       # Existing — status czytania (NIE RUSZAĆ):
       path("", ShelfListView.as_view()),
       path("<int:book_id>/", ShelfEntryView.as_view()),

       # New — kolekcje (Phase 2.3):
       path("me/collections/", MyShelvesView.as_view()),
       path("me/collections/<slug:shelf_slug>/", MyShelfDetailView.as_view()),
       path("me/collections/<slug:shelf_slug>/books/<int:book_id>/", MyShelfBookView.as_view()),
   ]
   ```
   **Uwaga o ścieżce:** kontrakt frontendu mówi `/me/shelves` (bez prefiksu) — w Django backend wszystko jest pod `/api/shelf/`. Realne ścieżki: `/api/shelf/me/collections/...`. Frontend wrapper (`src/lib/api/shelves.ts` w Svelte) mapuje `/me/shelves` → `/api/shelf/me/collections/`. **Decyzja:** używamy `me/collections/` **a nie** `me/shelves/` żeby unikać zderzenia URL z istniejącym `path("", ShelfListView)` reprezentującym "shelf entries" (status). Alternatywą byłby przemianowany prefiks całej app (`/api/collections/`), ale to wymaga zmiany w kliencie Vue jeszcze żyjącym. Wybieramy minimalną inwazję.

2. **Nowe widoki** — `shelf/views.py` (dodać do istniejącego pliku):
   - `MyShelvesView(generics.ListCreateAPIView)`:
     - `GET /api/shelf/me/collections/` — zwraca wszystkie półki zalogowanego usera (bez paginacji, decyzja grilling: <20 shelvy per user typowo).
     - `POST /api/shelf/me/collections/` — body: `{name, description?, is_public?}`. Slug generowany serwerowo. 201 → pełny obiekt.
     - Permissions: `IsAuthenticated`. Queryset filtrowany po `user=request.user`.
   - `MyShelfDetailView(generics.RetrieveUpdateDestroyAPIView)`:
     - `GET /api/shelf/me/collections/<slug>/` — szczegóły + lista książek (paginated lub flat — patrz `ShelfDetailSerializer` poniżej).
     - `PATCH` — edycja `name`, `description`, `is_public`. Zmiana `name` **NIE** regeneruje sluga (slug stable for URL). Manualna zmiana sluga poza MVP.
     - `DELETE` — usunięcie półki (CASCADE membershipy).
   - `MyShelfBookView(APIView)`:
     - `POST /api/shelf/me/collections/<slug>/books/<book_id>/` — dodaje book do półki. Idempotentne (`get_or_create`). 201 → `{}` (lub `{added: bool}`). 404 jeśli książka nie istnieje.
     - `DELETE /api/shelf/me/collections/<slug>/books/<book_id>/` — usuwa membership. 204 → `{}`. 404 jeśli nie był członkiem.

3. **Serializery** — `shelf/serializers.py` (dodać):
   ```python
   class ShelfSerializer(serializers.ModelSerializer):
       book_count = serializers.IntegerField(source="memberships.count", read_only=True)
       isPublic = serializers.BooleanField(source="is_public")
       createdAt = serializers.DateTimeField(source="created_at", read_only=True)
       updatedAt = serializers.DateTimeField(source="updated_at", read_only=True)

       class Meta:
           model = Shelf
           fields = ("id", "name", "slug", "description", "isPublic", "book_count", "createdAt", "updatedAt")
           read_only_fields = ("id", "slug", "book_count", "createdAt", "updatedAt")

   class ShelfDetailSerializer(ShelfSerializer):
       books = serializers.SerializerMethodField()

       class Meta(ShelfSerializer.Meta):
           fields = ShelfSerializer.Meta.fields + ("books",)

       def get_books(self, obj):
           # Plaska lista book preview {id, title, slug, cover_url, author}
           memberships = obj.memberships.select_related("book").prefetch_related("book__authors").order_by("position", "-added_at")
           return [self._book_preview(m.book) for m in memberships]

       def _book_preview(self, book):
           author = book.authors.first().name if book.authors.exists() else None
           return {
               "id": book.id,
               "slug": getattr(book, "slug", None),  # zakłada Phase 2.1 (slug na Book)
               "title": book.title,
               "author": author,
           }
   ```
   **Zależność od Phase 2.1:** `slug` na `Book` to scope Phase 2.1. Jeśli 2.1 nie jest jeszcze zmergowana podczas implementacji 2.3, używamy `getattr(book, "slug", None)` — fallback None. Po zmerge'owaniu 2.1 pole będzie zawsze obecne.

4. **Paginacja membershipów wewnątrz shelf detail** — w MVP: bez. Półka typowo <100 książek; serwujemy całą listę książek z poziomu detail endpoint. Jeśli profile w przyszłości urosną, dodamy `?page=` w osobnym etapie (osobny endpoint `/me/collections/<slug>/books/` zwracający `Paginated<Book>`).

### Non-goals (Obszar 2)

- Endpoint reorderowania pozycji (drag-and-drop) — pole `position` istnieje, UI nie używa
- Bulk add (`POST /me/collections/<slug>/books/` z `[bookId, bookId]`) — pojedyncze operacje wystarczą dla MVP
- Filtrowanie / search w obrębie półki — frontend filtruje po stronie klienta

## Obszar 3 — Publiczny endpoint `/u/:handle/shelves`

### Stan obecny

- `users/urls/users.py` rooteuje `/<str:username>/` → `UserProfileView`, `/<str:username>/followers/`, `/<str:username>/following/`
- `config/urls.py:8` montuje pod `/api/users/`
- Frontend kontrakt mówi `/u/:handle/shelves` (gdzie `handle == username`) → realna ścieżka `/api/users/<username>/shelves/`. Mapowanie `/u/:handle` → `/api/users/:username/` jest w gestii frontu (Svelte wrapper)
- `UserProfileView` (`users/views.py:127`) respektuje `profile_public` — gdy profil prywatny i requestujący to nie właściciel, zwraca 404

### Planowane zmiany

1. **Nowy widok `PublicUserShelvesView`** — `users/views.py` (lub osobny `shelf/views.py` jeśli wolimy domenowo — patrz Obszar 4 decyzja):
   ```python
   class PublicUserShelvesView(generics.ListAPIView):
       """
       Publiczne półki użytkownika o danym username.
       - Jeśli `profile_public=False` i request user != owner → 404 (analogicznie do UserProfileView).
       - Zwraca półki gdzie `is_public=True`, OWNER widzi też swoje prywatne.
       """
       permission_classes = (permissions.AllowAny,)
       serializer_class = ShelfSerializer
       pagination_class = None

       def get_queryset(self):
           username = self.kwargs["username"]
           target = get_object_or_404(User, username=username)
           if not target.profile_public and target != self.request.user:
               raise Http404("No User matches the given query.")
           qs = Shelf.objects.filter(user=target)
           if target != self.request.user:
               qs = qs.filter(is_public=True)
           return qs.order_by("-created_at")
   ```

2. **URL** — `users/urls/users.py` dodajemy:
   ```python
   path("<str:username>/shelves/", PublicUserShelvesView.as_view()),
   ```

3. **Decyzja co do follow endpointów** — `UserFollowView` i `FollowListView` **istnieją w kodzie** (`users/views.py:167-234`), bo były dodane wcześniej. Grilling-decision: follow system **WYCIĘTY z MVP frontendu**. Postępujemy:
   - **Backend zostawiamy** (kod działa, ma testy) — usunięcie wymagałoby migracji modelu `UserFollow` co jest niepotrzebne ryzyko
   - **NIE dodajemy** ich do dokumentacji frontendu Svelte; w UI brak przycisku follow
   - Decyzja "remove from API surface" potencjalnie w Phase 4 jeśli zostają nieużywane

### Non-goals (Obszar 3)

- Endpoint `/u/:handle/activity` — WYCIĘTY z MVP (grilling-decision: zastąpiony "Recently read" — patrz Obszar 4)
- 3-state visibility (friends/public/private) — `profile_public` jest boolem (już zaimplementowane)
- Publiczny endpoint pojedynczej półki `/u/:handle/shelves/<slug>/` — jeśli okaże się potrzebny w UI, dodamy w polish-pass; podstawa MVP to lista

## Obszar 4 — "Recently read" zamiast activity feed

### Stan obecny

- Frontend kontrakt mówił o `GET /u/:handle/activity` → `Paginated<ActivityItem>` (paginowany feed)
- Grilling-decision: activity feed WYCIĘTY, zamiast tego **"Recently read" 6 okładek** na profilu
- `ShelfEntry` ma `status=READ` + `finish_date` (`shelf/models.py:11,29`) — wystarczy do tej feature

### Planowane zmiany

1. **Endpoint `/u/:handle/recently-read/`** — nowy w `users/urls/users.py`:
   ```python
   path("<str:username>/recently-read/", PublicUserRecentlyReadView.as_view()),
   ```

2. **Widok** — w `users/views.py` (lub `shelf/views.py` jak Obszar 3):
   ```python
   class PublicUserRecentlyReadView(generics.ListAPIView):
       """6 ostatnio przeczytanych książek użytkownika (status=READ, sortowane po finish_date desc).
       Respektuje profile_public analogicznie do UserProfileView."""
       permission_classes = (permissions.AllowAny,)
       serializer_class = BookPreviewSerializer  # nowy lub reused z books
       pagination_class = None

       def get_queryset(self):
           username = self.kwargs["username"]
           target = get_object_or_404(User, username=username)
           if not target.profile_public and target != self.request.user:
               raise Http404("No User matches the given query.")
           return (
               Book.objects.filter(
                   shelf_entries__user=target,
                   shelf_entries__status=ShelfEntry.Status.READ,
               )
               .order_by("-shelf_entries__finish_date", "-shelf_entries__created_at")
               .distinct()[:6]
           )
   ```
   - Hardcoded limit 6 — UI tile grid. Konfigurowalne przez query `?limit=` poza MVP.
   - `finish_date` może być `NULL` (user nie podał) — fallback na `created_at`. Decyzja: `NULL`-y na końcu (Django: `nulls_last=True` w `OrderBy`).

3. **`BookPreviewSerializer`** — minimalny serializer książki (id, slug, title, author, cover_url jeśli istnieje). Decyzja: jeśli w `books/serializers.py` jest już lekki preview, reusujemy; jeśli nie — tworzymy `BookPreviewSerializer` w `books/serializers.py` (deal z Phase 2.1 — może już tam być).

### Non-goals (Obszar 4)

- Pełny activity feed (rated X, finished Y, added Z to shelf, follow) — wycięte
- Paginacja "recently read" — hardcoded 6
- Filtrowanie po dacie / per gatunek — UI tego nie potrzebuje

## Out of scope (dla Phase 2.3)

Wycięte na grilling-sessions z 2026-05-23 — wypisane explicite żeby agent implementacji nie nadinterpretował zakresu:

- **Activity feed** (`/u/:handle/activity` jako paginowana lista wydarzeń) — zastąpione przez "Recently read" 6 okładek (Obszar 4)
- **Follow system** (UI follow/unfollow przycisk + lista followers/following w SvelteKit) — backend kod (`UserFollow`, `UserFollowView`, `FollowListView`) zostawiamy, ale **nie konsumujemy** w Svelte MVP; usunięcie z API surface to potencjalna Phase 4
- **3-state visibility** profilu (friends/public/private) — `profile_public` jest boolem i tak zostaje
- **Endpoint reorderowania pozycji** w półce (drag-and-drop) — pole `position` w `ShelfMembership` istnieje na przyszłość, endpoint poza MVP
- **Współdzielenie półek między userami** (kolaboratywne kolekcje) — single-owner wystarczy
- **Bulk operations** (POST `books: [id1, id2]` jednym requestem) — pojedyncze operacje
- **Paginacja na liście shelvy** (`/me/collections/`) — typowo <20 per user, flat list
- **Paginacja membershipów wewnątrz shelf detail** — typowo <100, flat
- **Zmiana `ShelfEntry` modelu** (np. dodanie typu "favorited") — `ShelfEntry` zostaje nietknięty, kolekcje to osobna domena
- **Cross-book Character** współpraca z półkami (filtr "books containing X" w obrębie półki) — wycięte z MVP zgodnie z ADR-003 (Faza 3+)
