# Phase 2.5 — Character disambiguation (alias merge in-book)

**Data:** 2026-05-23
**Zakres:** `backend-django/analysis/` (model, views, serializers, urls, admin, testy), `backend-django/books/` (serializer/view — filtr aliasów), migracje, OpenAPI snapshot
**Stan wyjściowy:** Po Phase 2.0 (snapshot OpenAPI, CORS+cookies), Phase 2.1 (Books API: paginacja, `slug` na `Book`, `contains-character`) i **Phase 2.2 (AI extraction)** — `BookCharacter` ma już `slug`, `source`, `confidence`, `is_hidden`; istnieje `analysis/views.py`, `analysis/serializers.py`, `analysis/urls.py` z endpointami `AIExtractionTriggerView`, `AIExtractionRetrieveView`, `BookCharacterHideView`. `analyse_book` jest idempotentny (upsert po `slug`).

---

## Kontekst

NER (spaCy `en_core_web_trf`) wykrywa imiona postaci jako osobne tokeny — to samo entity występuje w tekście jako "Harry", "Harry Potter", "Mr. Potter", "Potter". Po `analyse_book` w bazie powstają **trzy osobne** rekordy `BookCharacter` opisujące tę samą osobę. Soft-delete `is_hidden` z Phase 2.2 pozwala ukryć fałszywie pozytywne encje, ale **nie scala** `mention_count` ani **nie przepisuje relacji** (`CharacterRelationship.from_character` / `to_character`) z duplikatów na canonical.

Phase 2.5 wprowadza **alias merge w obrębie jednej książki**:

1. **Pole `canonical`** (FK self) na `BookCharacter`. `canonical = NULL` oznacza "to jest canonical"; `canonical = <id>` oznacza "to jest alias wskazujący na tamtą encję".
2. **Endpoint `POST /api/books/:bookId/characters/:id/merge`** (admin-only) — body `{ "into": <characterId> }`. Przepisuje wszystkie `CharacterRelationship` z aliasu na canonical, sumuje `mention_count`, ustawia `canonical = into`, `is_hidden = True`.
3. **Filtr w listingu** — `BookDetailSerializer` zwraca tylko encje canonical (`canonical__isnull=True`, `is_hidden=False`) dla non-admin. Admin widzi wszystkie z badge'em.
4. **`slug` (z Phase 2.2) na canonical** — aliasy nie muszą mieć unique slug (mogą się powtarzać między aliasem a innym aliasem) bo i tak są ukryte, ale zachowujemy unique constraint dla prostoty (nikt nie tworzy aliasów ręcznie, merge zostawia istniejący slug).

Cel: admin po `analyse_book` przegląda listę charakterów w UI Svelte (Phase 2.8, prompt `06-ai-cast-panel.md`), zaznacza duplikaty i robi merge. Po merge'u API zwraca jedną canonical encję z poprawnym `mention_count` i kompletem relacji.

**Wymiar zakresu:** mały — 1 nowe pole, 1 nowy endpoint, jedna ścieżka serializacji do dostosowania. Mieści się w jednym PR (`phase/2.5-disambiguation`).

## Obszar 1 — Pole `canonical` na `BookCharacter`

### Stan obecny

- `analysis/models.py:4-18` — `BookCharacter` ma `book`/`name`/`description`/`mention_count`. Po Phase 2.2 dochodzi `slug`/`source`/`confidence`/`is_hidden`.
- Brak FK self do oznaczenia aliasów. Soft-delete `is_hidden` jedyna ścieżka "pozbycia się" duplikatu, ale tracimy `mention_count` i `CharacterRelationship` (zostają wiszące na ukrytej encji).
- `BookCharacter.Meta.constraints` (`analysis/models.py:12-15`) ma `UniqueConstraint(["name", "book"])` oraz (po 2.2) `UniqueConstraint(["slug", "book"])`. Te zostają.

### Planowane zmiany

1. **Migracja modelu** (`analysis/migrations/0005_bookcharacter_canonical.py`):
   - `canonical` — `ForeignKey("self", on_delete=models.SET_NULL, null=True, blank=True, related_name="aliases", limit_choices_to={"canonical__isnull": True})`.
     - `SET_NULL`: jeśli ktoś usunie canonical w Django Admin, alias staje się samodzielną canonical (lepsze niż kaskada usuwania aliasów).
     - `limit_choices_to`: w Django Admin formularz pokaże tylko encje canonical jako kandydatów na `canonical` aliasu (nie pozwala zrobić łańcucha alias→alias→canonical).
   - **Brak backfillu** — wszystkie istniejące rekordy mają `canonical = NULL` (czyli są canonical).
   - **Index**: `models.Index(fields=["book", "canonical"], name="bookchar_book_canonical_idx")` — accelerates filter `book=X, canonical__isnull=True` w listingu (najgorętsze query).
2. **`analysis/models.py` — `BookCharacter.clean()`**:
   - Walidacja: `canonical` nie może wskazywać na samego siebie (`if self.canonical_id == self.id: raise ValidationError(...)`).
   - Walidacja: `canonical.book_id == self.book_id` (alias i canonical muszą być w tej samej książce — alias merge cross-book jest WYCIĘTY z MVP, patrz Out of scope).
   - Walidacja: `canonical.canonical_id is None` (zapobiega tworzeniu łańcuchów alias→alias).
   - `save()` wywołuje `self.full_clean()` jeśli `canonical_id` jest ustawione (na save'ach bez merge'a nic się nie dzieje).
3. **`BookCharacter.is_canonical` (property)**: `return self.canonical_id is None`. Używane w serializerach i adminie dla czytelności.

### Non-goals (Phase 2.5)

- Cross-book canonical (jedna globalna `Character` z M2M do wielu `Book`) — to **Faza 3** z ADR-005 (nowy model `Character`, wymagałby przepisania całego data flow NER).
- Aliasy bez merge'a (admin ręcznie tworzy alias przed merge'em) — niepotrzebne, alias powstaje TYLKO przez `POST /merge/`.
- `also_known_as` jako lista stringów (bez powiązania z innym `BookCharacter`) — odrzucone na rzecz pełnego FK, bo i tak chcemy scalać `CharacterRelationship`.

## Obszar 2 — Endpoint `POST /api/books/:bookId/characters/:id/merge`

### Stan obecny

- `analysis/urls.py` (po Phase 2.2) zawiera 3 routes: `ai/extract`, `ai/extraction`, `characters/:id/hide`.
- Brak endpointu merge. Admin musiałby ręcznie w Django Admin: (1) zmienić wszystkie `CharacterRelationship.from_character` / `to_character` z aliasu na canonical, (2) dodać `alias.mention_count` do `canonical.mention_count`, (3) ustawić `alias.is_hidden = True`. Praktycznie niewykonalne dla 5+ duplikatów na książkę.

### Planowane zmiany

1. **`analysis/views.py` — `BookCharacterMergeView`**:
   - `POST /api/books/<int:book_id>/characters/<int:character_id>/merge/`.
   - Permissions: `IsAdminUser` (hardcoded, zgodnie z grilling — wszystkie endpointy mutujące AI są admin-only).
   - Walidacja request body (`MergeRequestSerializer`):
     - `into` — int, wymagane, ID encji canonical w tej samej książce.
     - 400 jeśli body niepoprawne.
   - Walidacja stanu:
     - `BookCharacter.objects.get(id=character_id, book_id=book_id)` → 404 jeśli brak (źródło merge'a).
     - `BookCharacter.objects.get(id=body["into"], book_id=book_id)` → 400 (`"target character not in this book"`) jeśli `book_id` się nie zgadza, 404 jeśli encja nie istnieje wcale.
     - 400 jeśli `character_id == body["into"]` (`"cannot merge character into itself"`).
     - 400 jeśli źródło ma `canonical_id is not None` (`"source is already an alias; merge the canonical instead"`).
     - 400 jeśli target ma `canonical_id is not None` (`"target is an alias; provide the canonical id"`).
   - **Transakcja merge** (`@transaction.atomic`):
     1. `CharacterRelationship.objects.filter(from_character=source).update(from_character=target)` — przepisz wszystkie out-relacje.
     2. `CharacterRelationship.objects.filter(to_character=source).update(to_character=target)` — przepisz wszystkie in-relacje.
     3. **Deduplikacja relacji** — po update kroków 1-2 może powstać duplikat naruszający `UniqueConstraint(["from_character", "to_character", "book"])` (`analysis/models.py:94-98`). Strategia:
        - Przed `update()` w krokach 1-2 zbierz `existing = set((rel.from_character_id, rel.to_character_id, rel.relation_type) for rel in CharacterRelationship.objects.filter(book=book))`.
        - Dla każdej relacji ze źródła: jeśli po przepisaniu (`from=target` lub `to=target`) krotka `(from, to, type)` byłaby w `existing` → usuń tę relację zamiast `update`'ować. Inaczej `update`.
        - Implementacja: iteracja po `CharacterRelationship.objects.filter(Q(from_character=source) | Q(to_character=source))` — Python-level dedup, nie bulk update (mała liczba relacji per książkę, max ~50, więc wydajność nie jest problemem).
     4. `target.mention_count = F("mention_count") + source.mention_count; target.save(update_fields=["mention_count"])`.
     5. `source.canonical = target; source.is_hidden = True; source.save(update_fields=["canonical", "is_hidden"])`.
     6. **Aliasy aliasu (edge case)** — jeśli ktoś próbuje scalić canonical X w Y, ale X ma już swoje aliasy `[A1, A2]` — krok walidacji wyżej już to blokuje (`source.canonical_id is not None`). Ale **odwrotny** scenariusz: source = canonical X z aliasami A1, A2. Po merge'u X→Y, A1/A2 wciąż wskazują na X. Strategia: `BookCharacter.objects.filter(canonical=source).update(canonical=target)` przed krokiem 5. Wtedy A1, A2 dostają nowy canonical Y, a sam X dostaje `canonical=Y` w kroku 5.
   - Response: 200 z `BookCharacterSerializer(target).data` — frontend dostaje zaktualizowaną canonical encję z nowym `mention_count`.
2. **`analysis/serializers.py` — `MergeRequestSerializer`** (lub inline `serializers.Serializer` w view):
   ```python
   class MergeRequestSerializer(serializers.Serializer):
       into = serializers.IntegerField(min_value=1)
   ```
3. **`analysis/urls.py`** — dodać route:
   ```python
   path("books/<int:book_id>/characters/<int:character_id>/merge/", BookCharacterMergeView.as_view()),
   ```
4. **Brak endpointu undo / unmerge** — by design (decyzja grilling: merge jest **nieodwracalny przez API**). Admin może w Django Admin ręcznie ustawić `alias.canonical = None`, ale `mention_count` i relacje już nie wrócą do oryginalnego stanu. To trade-off za prostotę.

### Non-goals (Phase 2.5)

- **`POST /merge/preview`** (dry-run pokazujący ile relacji zostanie scalonych) — Phase 4.
- **Bulk merge** (`POST /books/:id/characters/merge-batch/`) — Phase 4. MVP: po jednym merge na request.
- **LLM-assisted merge suggestions** (`GET /books/:id/characters/duplicates/`) — WYCIĘTE z MVP (grilling: czysto manualne).
- **Auto-merge na podstawie similarity** (Levenshtein, embeddings) — WYCIĘTE z MVP.

## Obszar 3 — Filtr canonical w listingu

### Stan obecny

- Po Phase 2.2 `BookDetailSerializer.to_representation` (`books/serializers.py:107-110`) filtruje `instance.characters.filter(is_hidden=False)` dla non-admin.
- `AIExtractionSerializer` (`analysis/serializers.py`, dodany w Phase 2.2) zwraca `BookCharacterSerializer(book.characters.all(), many=True)` — admin widzi wszystkie.

### Planowane zmiany

1. **`books/serializers.py:107-110` — `BookDetailSerializer.to_representation`**:
   - Zmiana filtra dla non-admin: `instance.characters.filter(is_hidden=False, canonical__isnull=True)`.
   - Powód: aliasy mają `is_hidden=True` PO merge'u (krok 5 transakcji), ale dla bezpieczeństwa filtrujemy też po `canonical__isnull=True` — jeśli admin ręcznie odhide'uje alias w Django Admin, nadal nie pokazujemy go w publicznym listingu.
   - Admin: bez zmian (widzi wszystkie, włącznie z aliasami i hidden).
2. **`books/views.py:42-49` — `BookRetrieveView`**:
   - Phase 2.2 zakłada `Prefetch("characters", queryset=BookCharacter.objects.filter(is_hidden=False))` dla non-admin.
   - Phase 2.5: rozszerz queryset prefetch'a o `canonical__isnull=True`:
     ```python
     Prefetch(
         "characters",
         queryset=BookCharacter.objects.filter(is_hidden=False, canonical__isnull=True),
         to_attr="visible_characters",
     )
     ```
   - Serializer używa `instance.visible_characters` zamiast `instance.characters.all()` dla non-admin (oszczędza query).
3. **`analysis/serializers.py` — `AIExtractionSerializer.characters`**:
   - Admin widzi WSZYSTKO (`book.characters.all()`) — w tym aliasy z `canonical_id` i `is_hidden=True`. UI Svelte (Phase 2.8) pokaże je z badge'em "alias of X".
   - `BookCharacterSerializer` musi zwracać też `canonicalId` (camelCase: `canonicalId`), żeby UI wiedział do kogo wskazuje alias.

### Planowane zmiany — `BookCharacterSerializer`

`books/serializers.py:8-13` (i / lub jego odpowiednik w `analysis/serializers.py`, w zależności od ustaleń Phase 2.2):

```python
class BookCharacterSerializer(serializers.ModelSerializer):
    mentionCount = serializers.IntegerField(source="mention_count")
    isHidden = serializers.BooleanField(source="is_hidden")
    canonicalId = serializers.IntegerField(source="canonical_id", allow_null=True)

    class Meta:
        model = BookCharacter
        fields = (
            "id", "name", "slug", "description",
            "mentionCount", "source", "confidence",
            "isHidden", "canonicalId",
        )
```

### Non-goals (Phase 2.5)

- Pokazywanie aliasów w publicznym listingu (`GET /books/:id`) z badge'em "(also known as Harry)" — to UI decision dla Phase 2.8 (Svelte), backend dostarcza canonical + aliasy oddzielnie tylko dla admina.
- Endpoint `GET /books/:id/characters/:slug/aliases/` (lista aliasów canonical) — niepotrzebny w MVP, admin widzi to w Django Admin (FK `aliases` via `related_name`).

## Obszar 4 — Django Admin integracja

### Stan obecny

- `analysis/admin.py:13-17` — `BookCharacterAdmin` pokazuje `name`, `book`, `mention_count`. Phase 2.2 dodaje `slug`, `source`, `is_hidden`.
- Brak filtra po `canonical` / aliasach. Brak akcji merge w adminie (admin musi używać HTTP API).

### Planowane zmiany

1. **`analysis/admin.py` — `BookCharacterAdmin`**:
   - `list_display` dodać `canonical` (Django renderuje jako link do canonical, lub `—` dla canonical samych).
   - `list_filter` dodać `("canonical", admin.EmptyFieldListFilter)` — filtr "Aliases only" / "Canonical only".
   - `raw_id_fields = ("canonical",)` — dla książek z 50+ charakterami dropdown by się rozjechał.
   - Akcja masowa **`unmerge_aliases`** (admin-only escape hatch dla błędnych merge'ów):
     - Dla zaznaczonych aliasów: `canonical = None`, `is_hidden = False`. **Nie odwraca** przepisania relacji ani sumy `mention_count` (admin musi to skorygować ręcznie). Komunikat warning po wykonaniu: `"Aliases unmerged. Note: relations and mention counts were NOT restored — fix manually if needed."`.
2. **Brak akcji `merge_selected` w adminie** — HTTP endpoint `POST /merge/` jest jedynym sposobem na merge (transakcja, deduplikacja relacji, walidacja). Admin chcąc scalić w UI używa Svelte AICastPanel albo Django Admin → przycisk "Merge via API" (link do dokumentacji). Powód: duplikacja logiki merge w dwóch miejscach to ryzyko bug parity.

### Non-goals (Phase 2.5)

- Custom admin view z UI merge'a (przycisk "Merge X into Y" w detail page) — Phase 4, jeśli będzie potrzebne.
- Audit log merge'ów (`MergeLog` model) — Phase 4. Na razie commit message w git + Django Admin LogEntry (out-of-box) wystarczy do debugowania.

## Out of scope (dla Phase 2.5)

Decyzje grilling-sessions z 2026-05-23 explicite wycinają z MVP:

- **Cross-book Character disambiguation** — alias merge działa TYLKO w obrębie jednej książki. Cross-book to **Faza 3** z nowym ADR-005 (model `Character` jako globalna encja + M2M z `BookCharacter`). ADR-003 (encje per-book) pozostaje w mocy dla MVP.
- **LLM-assisted merge suggestions** (`GET /books/:id/characters/duplicates/`) — czysto manualne. Admin samodzielnie identyfikuje duplikaty w UI Svelte (Phase 2.8) lub Django Admin.
- **Auto-merge na podstawie similarity** (Levenshtein, fuzzy match, embeddings) — wycięte. Wymagałoby tuning'u progu i ryzykuje fałszywe merge.
- **Undo merge przez API** — by design, merge jest **nieodwracalny** przez HTTP. Admin może w Django Admin użyć akcji `unmerge_aliases`, ale to ustawia tylko `canonical=None`/`is_hidden=False` — `mention_count` i przepisane `CharacterRelationship` **nie wracają** do stanu pre-merge.
- **Bulk merge endpoint** (`POST /books/:id/characters/merge-batch/`) — Phase 4. MVP: jeden merge na request.
- **Merge preview / dry-run** (`POST /merge/preview` zwraca ile relacji zostanie scalonych, czy będą duplikaty) — Phase 4.
- **Audit log merge'ów** (kto, kiedy, source→target) — Phase 4. Django Admin LogEntry (out-of-box) wystarczy do debugowania.
- **Aliasy bez merge'a** (admin tworzy alias ręcznie przed merge'em jako "manual entry") — niepotrzebne. Alias powstaje WYŁĄCZNIE jako wynik merge'a.
- **Pokazywanie aliasów w publicznym `GET /books/:id`** z badge'em "also known as" — UI decision dla Phase 2.8. Backend nie wystawia aliasów publicznie; admin widzi je przez `GET /ai/extraction/`.
- **Custom merge UI w Django Admin** (przycisk "Merge X into Y" na detail page) — Phase 4. Admin używa HTTP endpointu via curl / Postman / Svelte UI.
- **Otwarcie endpointu `merge` zalogowanym non-admin userom** (np. autorom książki) — hardcoded `IsAdminUser` (zgodnie z grilling: wszystkie endpointy mutujące AI są admin-only do Fazy 3).
