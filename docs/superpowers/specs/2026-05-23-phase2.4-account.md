**Data:** 2026-05-23
**Zakres:** backend-django/users/, backend-django/config/settings/, infra/docker-compose.*.yml
**Stan wyjściowy:** `User` (`users/models.py:26`) ma `email`, `username`, `bio`, `avatar_url` (URLField), `profile_public`, `is_active`. Istnieją: `GET/PUT/PATCH /api/users/me/` (`UserSettingsView`, `users/views.py:145`), `PATCH /api/users/me/visibility/` (`UserVisibilityView`, `users/views.py:155`), `GET /api/users/<username>/` (`UserProfileView`, `users/views.py:127`). **Brak** endpointów: zmiany hasła, zmiany emaila, multipart avatar upload, eksportu GDPR, usunięcia konta. `display_name` i `handle` nie istnieją (frontend wymaga ich jako pól osobnych od `username`). ## Kontekst

Phase 2.4 dostarcza kompletne API zarządzania kontem wymagane przez SvelteKit frontend (`svelte(wideframe)/handoff/03-api-contract.md`). Obecne API ma kilka niedostatków blokujących migrację:

1. **Brak rozdziału `username` (login handle) od `display_name`** — frontend pokazuje "Jan Kowalski" w UI, ale URL profilu używa `/u/jankowalski`. Obecny model ma tylko `username` używany w obu rolach.
2. **Avatar jako URLField** — wymaga zewnętrznego hostingu obrazka po stronie usera. Frontend zakłada multipart upload (`PATCH /users/me/avatar`), Django serwuje plik z `MEDIA_ROOT`.
3. **Brak zmiany hasła/emaila** — krytyczne operacje konta, których obecnie nie da się wykonać przez API (tylko Django admin).
4. **GDPR** — wymóg formalny dla każdej aplikacji przechowującej dane osobowe; brak endpointu eksportu i usuwania konta.
5. **`UserSettings` jako wydzielony zasób** — frontend oczekuje `GET /users/me` zwracającego `User & { settings: UserSettings }` oraz `PATCH /users/me/settings`. Obecnie wszystko jest płaskie w `UserSettingsSerializer`.

Reszta API publicznego profilu (`/u/:handle`) jest blisko gotowa — wymaga przepięcia z `username` na nowy `handle` jako lookup field i respektowania `profile_public`.

**Powiązane decyzje grilling-sessions (2026-05-23):**
- Visibility: bool `profile_public` (już w modelu) — nie 3-state. Bez zmian.
- Activity feed wycięty — endpoint profilu publicznego nie zwraca wydarzeń, tylko statyczne pola + (w Phase 2.3) shelfy publiczne, w Phase 2.1 "Recently read" derywowane z `ShelfEntry`.
- Follow system wycięty z MVP — `UserFollow` (`users/models.py:45`) **zostaje w modelu** (już istnieje, działa, ma testy), ale endpointy `/users/<username>/follow/`, `/followers/`, `/following/` **zostają undocumented** i nie są częścią kontraktu Svelte. Decyzja resurekcji/usunięcia — Phase 3+.
- Email verification: **out of scope Phase 2.4**. `PATCH /users/me/email` zmienia email **bezpośrednio** (po weryfikacji `current_password`). Token weryfikacyjny + wysyłka maila to Phase 3.
- GDPR export: synchroniczny ZIP z JSON; bez Celery, bez `download_url` z TTL — odpowiedź to bezpośredni `Content-Type: application/zip` stream (frontend wymusza pobranie). Pole `download_url` w kontrakcie odpada — zamiast tego endpoint zwraca plik.

## Obszar 1 — Refaktor modelu `User`: `handle` + `display_name`

### Stan obecny

- `User.username` (`users/models.py:28`): `CharField(max_length=30, unique=True)`, walidowany regexem `^[a-z]{3,30}$` (`users/serializers.py:13`)
- Pełni rolę zarówno identyfikatora w URL (`/u/<username>/`) jak i nazwy wyświetlanej w UI
- `USERNAME_FIELD = "email"` (login po emailu) — `username` nie jest używany do logowania
- Pole `display_name` **nie istnieje**

### Planowane zmiany

1. **Dodanie pola `display_name`** — `users/models.py`:
   ```python
   display_name = models.CharField(max_length=80, blank=True, default="")
   ```
   - Wolna forma (spacje, znaki diakrytyczne, emoji dozwolone)
   - Jeśli puste — UI fallback na `handle` (logika frontendu, nie backendu)
2. **Zmiana nazwy `username` → `handle`** — semantyczne dopasowanie do kontraktu Svelte (`/u/:handle`). **Bez zmiany typu i walidacji** (`^[a-z]{3,30}$`, unique). Migracja: `rename_field` w jednej migracji.
   - Aktualizacja `USERNAME_FIELD` zostaje na `email` (nic się nie zmienia w logice loginu)
   - Aktualizacja `REQUIRED_FIELDS = ["handle"]`
   - Aktualizacja wszystkich referencji: `users/admin.py`, `users/serializers.py`, `users/views.py`, `users/urls/users.py`, testy
3. **Migracja danych** — `0007_rename_username_to_handle.py`:
   ```python
   migrations.RenameField("User", "username", "handle")
   migrations.AddField("User", "display_name", ...)
   # data migration: display_name = handle dla istniejących userów
   ```
4. **Serializer rozdział** — `users/serializers.py`:
   - `RegisterSerializer`: pole nazywa się teraz `handle` (regex bez zmian), opcjonalne `display_name` przy rejestracji (jeśli puste — defaults `""`)
   - `UserProfileSerializer`: zwraca `handle`, `display_name`, `bio`, `avatar_url`, `member_since`, `profile_public` (nie `username`, nie `avatarUrl` camelCase — patrz Obszar 7)

### Wpływ na istniejące API

- **Breaking:** `RegisterSerializer` pole `username` → `handle` w request body. **Vue umiera w 2.6** — akceptowalne.
- **Breaking:** `/api/users/<username>/` → `/api/users/<handle>/` (oba używają tej samej kolumny DB; tylko nazwa parametru URL/dokumentu się zmienia)
- Endpoint `/u/:handle` (frontend) jest **nowym aliasem** — patrz Obszar 6

## Obszar 3 — `PATCH /users/me/password` — zmiana hasła

### Stan obecny

Brak endpointu. Reset hasła możliwy tylko przez `manage.py changepassword`.

### Planowane zmiany

1. **Nowy view** `PasswordChangeView` (`users/views.py`):
   - `PATCH /api/users/me/password/`
   - Wymaga uwierzytelnienia
   - Body: `{ "current_password": "...", "new_password": "..." }`
   - Walidacja: `request.user.check_password(current_password)` — jeśli false, `400` z `{"current_password": ["Invalid"]}`
   - `validate_password(new_password)` używając Django built-in validators (na razie tylko `MinimumLengthValidator`, base.py:74)
   - `request.user.set_password(new_password)` + `save()`
   - **Token rotation:** po zmianie hasła wszystkie istniejące refresh tokeny dla usera są blacklistowane (wymusza re-login na innych urządzeniach). Implementacja: `OutstandingToken.objects.filter(user=request.user).update(...)` + `BlacklistedToken.objects.bulk_create(...)` — lub prostsze: iterate i `.blacklist()`
   - Response: `204 No Content` (kontrakt mówi `{}` — dopuszczalne `200 {}` lub `204` — wybierzemy `204` bo nie ma payloadu)
   - **Re-issue cookies** dla bieżącej sesji: po blacklist iss new refresh+access dla `request.user` i set cookies w response — żeby user nie został wylogowany w przeglądarce z której zmienia hasło
2. **Throttling:** `user_password_change` scope, np. `5/hour`
3. **Serializer** — `PasswordChangeSerializer` w `users/serializers.py`:
   - Field `current_password`, `new_password` (write-only, min_length=6, max_length=72)

### Pliki

- `users/views.py` — nowy `PasswordChangeView`
- `users/serializers.py` — `PasswordChangeSerializer`
- `users/urls/users.py` — `path("me/password/", PasswordChangeView.as_view())`
- `config/settings/base.py:99` — dodać `"user_password_change": "5/hour"` do `DEFAULT_THROTTLE_RATES`

## Obszar 9 — `DELETE /users/me` — usunięcie konta

### Stan obecny

Brak. Konto można usunąć tylko przez Django admin.

### Planowane zmiany

1. **Nowy endpoint w `UserMeView`** (Obszar 2) — `delete()`:
   - `DELETE /api/users/me/`
   - Body: `{ "current_password": "..." }` (potwierdzenie — DELETE z body jest niestandardowe ale akceptowalne; alternatywa: query param, ale hasło w URL = zły pomysł)
   - Walidacja: `check_password(current_password)` — jeśli false, `403`
   - **Strategia usunięcia:** **hard delete** (`user.delete()`). Cascade usunie: `ShelfEntry` (FK CASCADE), `Review` (FK CASCADE), `UserFollow` (z obu stron), `OutstandingToken`/`BlacklistedToken`.
     - **Powód:** soft-delete (`is_active=False`) skomplikuje query (każdy `User.objects.filter(...)` musiałby filtrować `is_active=True`) i nie chroni przed niczym wartościowym dla hobby projektu. Hard delete = pełna zgodność z GDPR "right to erasure".
   - **Side effect:** `Book` referencje? **Nie** — `Book` nie ma FK do `User`. `Review.user` ma CASCADE.
   - **Co zostaje:** zero śladów po userze poza:
     - `Book.text` (nigdy nie zawierało user data)
     - `BookCharacter`/`Place`/`Organization` (per-book, nie per-user)
   - **Cookies:** `clear_jwt_cookies(response)` (analogicznie do `LogoutView`, `users/views.py:99`)
   - Response: `204 No Content`
2. **Throttling:** `user_delete` scope, np. `3/hour` (rzadkie, ale spam ochrona przy próbach brute-force hasła)

### Pliki

- `users/views.py` — `UserMeView.delete()` (lub osobny `AccountDeleteView` — preferujemy osobny dla jasności permission/throttle)
- `users/serializers.py` — `AccountDeleteSerializer` (`current_password`)
- `users/urls/users.py` — endpoint integrated z `me/` (lub `me/delete/` — wybieramy `me/` z metodą DELETE dla REST-owości)
- `config/settings/base.py:99` — throttle scope

## Out of scope (dla Phase 2.4 — explicite z grilling-sessions)

- **Email verification flow** (token wysyłany, klik linka, weryfikacja, dopiero wtedy zmiana emaila) — Phase 3. Phase 2.4 zmienia email natychmiast po potwierdzeniu hasła + wysyła informacyjny mail na stary adres.
- **Async data export** (Celery task + storage `download_url` z TTL) — synchroniczny stream wystarczy dla hobby projektu z max kilkuset wpisów per user. Aktualizacja kontraktu API w handoffie.
- **Soft-delete konta** (`is_active=False`, "30-day grace period") — hard delete, GDPR compliant, prościej.
- **3-state visibility** (private/friends/public) — `profile_public` bool zostaje. Friends wymaga follow system jako pierwowzór.
- **Activity feed** na profilu publicznym — frontend renderuje "Recently read" z `ShelfEntry` (kontrakt obsługiwany w Phase 2.1, nie 2.4).
- **Follow system endpointy** (`/users/<handle>/follow/`, `/followers/`, `/following/`) — model `UserFollow` zostaje, endpointy **zostają w kodzie** bo działają i mają testy, ale **nie są w OpenAPI snapshot** ani w handoffie Svelte. Decyzja eksponowania/usunięcia: Phase 3.
- **Avatar resize/normalizacja server-side** — frontend resizuje przed uploadem; backend tylko waliduje rozmiar+format+wymiary.
- **Reset hasła przez email ("forgot password")** — Phase 3 razem z email verification (oba potrzebują tego samego email flow).
- **2FA / TOTP** — Phase 4+ (jeśli kiedykolwiek).
- **OAuth / social login** — out of scope całego projektu (single-tenant hobby).
- **Refaktor camelCase → snake_case w pozostałych appkach** (books, shelf, reviews) — tylko `users/serializers.py` w Phase 2.4; reszta razem z 2.1/2.2/2.3.
- **Caddyfile update dla `/media/*` w prod** — dokumentowane w spec, ale faktyczna konfiguracja deploy w roadmap pkt 1.
- **Per-user notification preferences (email digest, etc.)** w `UserSettings` — Phase 2.4 daje strukturę `settings: {profile_public}`; rozszerzenia w Phase 3+ jako dodanie pól.
