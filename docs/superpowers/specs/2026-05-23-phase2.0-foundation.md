# Phase 2.0 — Foundation for Svelte Migration

**Data:** 2026-05-23
**Zakres:** backend-django/, frontend/, infra/
**Stan wyjściowy:** Backend gotowy do rozszerzeń (testy ✅, lint ✅, prod hardening ✅ po c2f9d2e). Vue frontend działa. Brak strategii testów kontraktowych dla API. JWT cookies skonfigurowane dla same-origin.

---

## Kontekst

Phase 2 (backend extensions) + Phase 3 (Svelte frontend) wymagają fundamentu który adresuje trzy problemy ujawnione w grilling/review:

1. **Brak kontraktu API jako test** — Svelte agent buduje frontend na podstawie `docs/handoff/03-api-contract.md` (statyczny dokument). Bez wykonywalnego kontraktu rozjazd backend ↔ frontend wykrywany dopiero w runtime.
2. **CORS i cross-origin JWT cookies nieskonfigurowane** — Svelte hostowany na osobnej origin (dev: `localhost:5174`, prod: `app.storyshelf.example.com` vs API `api.storyshelf.example.com`) wymaga `SameSite=None; Secure` na cookies + `django-cors-headers` z explicit allowlist. Obecny setup zakłada same-origin (Caddy serwuje Vue + API z tej samej domeny).
3. **Vue removal policy niejasna** — decyzja: Vue umiera po migracji, brak okresu przejściowego. Trzeba spisać kiedy, jak, i co konkretnie usunąć żeby nie zostawić martwego kodu.

To jest **fundament** — wszystkie kolejne specs (2.1-2.5) zakładają że te trzy problemy są rozwiązane.

## Obszar 1 — Kontrakt API jako test (OpenAPI snapshot)

### Stan obecny

- `drf-spectacular` zainstalowany, generuje schemat na `/api/schema/`
- Brak snapshot-testu który łapie nieintencjonalne zmiany kontraktu
- Brak generowania typów TS po stronie frontu (Vue używa ręcznie pisanych typów)
- Handoff `02-data-model.md` definiuje typy TS niezależnie od backendu

### Planowane zmiany

1. **Snapshot test OpenAPI** — `config/tests/test_openapi_schema.py`:
   - Generuje schemat przez `SpectacularAPIView.as_view()` (in-process)
   - Porównuje z plikiem `docs/api/openapi.yml` zacommittowanym do repo
   - Fail jeśli rozjazd; instrukcja regeneracji w error message
2. **CI step** — `make verify` uruchamia test schematu; PR który zmienia API **musi** zaktualizować `openapi.yml`
3. **Skrypt regeneracji** — `infra/scripts/regenerate-openapi.sh` jako utility (single command: `uv run python manage.py spectacular --file ../docs/api/openapi.yml --format openapi-yaml`)
4. **README sekcja** — w `backend-django/README.md` (lub w `docs/api/README.md`) jak regenerować i jak Svelte konsumuje (`openapi-typescript` w Phase 3)

### Non-goals (Phase 2.0)

- Pełne mock-server testowanie (Prism/Schemathesis) — nadmiar dla hobby projektu
- Versioning API (`/api/v1/`) — Vue umiera, brak konsumentów historycznych

## Obszar 2 — CORS + cross-origin JWT cookies

### Stan obecny

- `users/cookie_auth.py:35,44` ustawia JWT w HttpOnly cookies (ADR-001) z **hardcoded** `samesite="Lax"`
- `secure=` jest env-driven (`JWT_COOKIE_SECURE` settings)
- Endpoint loginu: `POST /api/auth/login/` (`users/urls/auth.py:7`)
- `django-cors-headers` **już zainstalowane** (`pyproject.toml`) i `corsheaders.middleware.CorsMiddleware` **już w MIDDLEWARE** (`config/settings/base.py`)
- `config/settings/dev.py` ma `CORS_ALLOW_ALL_ORIGINS = True` — wszystkie origins akceptowane
- `config/settings/prod.py` ma `CORS_ALLOWED_ORIGINS` env-driven oraz `CSRF_TRUSTED_ORIGINS` (wymagane, fail-fast jeśli brak)
- **Brak** `CORS_ALLOW_CREDENTIALS = True` — wymagane dla cookies cross-origin

### Planowane zmiany

1. **Refactor `users/cookie_auth.py`** — `samesite` z hardcoded `"Lax"` na env-driven (czytane z settings `JWT_COOKIE_SAMESITE`, default `"Lax"`):
   - `base.py`: `JWT_COOKIE_SAMESITE = os.getenv("JWT_COOKIE_SAMESITE", "Lax")`
   - Refactor `set_jwt_cookies()` żeby brało `samesite` z settings, nie z literału
2. **Settings cross-origin defaults:**
   - `base.py`: `CORS_ALLOW_CREDENTIALS = True` (wymagane dla cookies), `JWT_COOKIE_SAMESITE` (default `"Lax"`)
   - `dev.py`: **zmiana** — z `CORS_ALLOW_ALL_ORIGINS = True` na explicit `CORS_ALLOWED_ORIGINS = ["http://localhost:5174", "http://localhost:5173"]`. **Powód:** `CORS_ALLOW_ALL_ORIGINS=True` jest niekompatybilne z `CORS_ALLOW_CREDENTIALS=True` (django-cors-headers ignoruje credentials w trybie wildcard origin). Cookie auth dev wymaga explicit allowlist.
   - `prod.py`: ustawić `JWT_COOKIE_SAMESITE = "None"` (override z env: `JWT_COOKIE_SAMESITE=None` jako os env). `CORS_ALLOWED_ORIGINS` env-driven już jest. Opcjonalnie: `JWT_COOKIE_DOMAIN` env-driven (`.storyshelf.example.com`) — domyślnie pomijamy (host-only cookie wystarczy).
3. **Update istniejących testów** — `users/tests/test_cookie_auth.py:66` asercja `samesite="Lax"` — przepiąć na sprawdzanie ze settings (`from django.conf import settings; expected = settings.JWT_COOKIE_SAMESITE`)
4. **Nowy test integracyjny** — `users/tests/test_cors_jwt.py`:
   - Preflight OPTIONS `/api/auth/login/` z `Origin: http://localhost:5174` zwraca `Access-Control-Allow-Origin` + `Access-Control-Allow-Credentials: true`
   - Login response Set-Cookie ma `SameSite` zgodny z settings (dev: `Lax` bez `Secure`; testowany override settings dla symulacji prod: `None` + `Secure`)
5. **Caddy config (infra)** — opcjonalny: jeśli zostajemy przy split-domain w prod, Caddyfile potrzebuje osobnych bloków dla `api.` i `app.` subdomen. Decyzja deploy do oddzielnego etapu (roadmap pkt 1), tu tylko backend gotowy

### Non-goals (Phase 2.0)

- Faktyczna konfiguracja domen produkcyjnych — zależy od decyzji deployu (roadmap pkt 1)
- HTTPS w dev (mkcert/local CA) — `Secure` cookie zostaje wyłączone w dev settings
- CSRF refactor — DRF + SessionAuthentication wyłączone, JWT nie potrzebuje CSRF tokens (Cookie + SameSite=None wymaga ostrożności, ale nasz `JWTCookieAuthentication` nie używa Django session)

## Obszar 3 — Vue removal policy

### Stan obecny

- `frontend/` zawiera Vue 3 SPA: `src/`, `tests/`, `vite.config.js`, `package.json`
- Docker compose ma serwis `vue` (Nginx serwujący build)
- `infra/caddy/` rooteuje `/` → vue, `/api/*` → django
- `Makefile` ma cele odwołujące się do Vue (`make dev-up`, `make build-frontend`?)
- CI workflow buduje obraz Vue

### Polityka usuwania

**Moment usunięcia:** Phase 3.0 — pierwsza akcja po zakończeniu Phase 2. Vue zostaje **w pełni funkcjonalny** przez całe Phase 2; znika **w jednym commicie** na początku Phase 3.

**Co zostaje usunięte (commit `chore: remove vue frontend`):**

| Ścieżka | Akcja |
|---|---|
| `frontend/` | usunąć całość (`rm -rf`) |
| `infra/docker-compose.dev.yml`, `prod.yml` — serwis `vue` | usunąć blok |
| `infra/caddy/Caddyfile` (lub równoważny) — root `/` → vue | tymczasowo `respond "Migration in progress"` albo redirect na nowy Svelte build |
| `Makefile` — cele `*-frontend`, `dev-up` w części Vue | dostosować |
| `.github/workflows/ci.yml` — Vue build step | usunąć |
| `CLAUDE.md` (`## Komendy → Frontend`) | aktualizacja (link do nowego repo Svelte) |
| `docs/ARCHITECTURE.md` — sekcja "Frontend" wzmianki Vue | aktualizacja |

**Co zostaje:**
- `backend-django/` w pełni nietknięty (CORS już rozszerzone — przygotowane na Svelte z innej domeny)
- API kontrakt (Vue umierający łamie żadnych konsumentów którymi powinniśmy się martwić)
- `docs/decisions/ADR-001-jwt-httponly-cookies.md` — JWT cookies zostają, Svelte ich używa

### Phase 2.0 — co robimy z polityką teraz

- **Tylko spisanie polityki** w `docs/decisions/ADR-004-vue-removal.md` (nowy ADR)
- **Nie** kasujemy Vue w Phase 2.0; kasowanie to pierwsza akcja Phase 3.0
- ADR jest dokumentem decyzyjnym + check-listą do wykonania w Phase 3.0

### Non-goals (Phase 2.0)

- Faktyczne usunięcie Vue — to Phase 3.0
- Migracja Svelte repo setup — to Phase 3.0
- Sprzątanie historii git Vue (squash, tag) — niepotrzebne, hobby projekt
