# Phase 2.1 — Books API extensions

**Data:** 2026-05-23
**Zakres:** `backend-django/books/`, `backend-django/reviews/`, `backend-django/config/`
**Stan wyjściowy:** Phase 2.0 zakończona (OpenAPI snapshot, CORS+cookies cross-origin, ADR-004 Vue removal). `GET /api/books/` zwraca **płaską listę** bez paginacji (`pagination_class = None`, `books/views.py:18`). `GET /api/reviews/?book_id=` używa query param zamiast scope w URL (`reviews/views.py:34`). Brak slugu na `Book`, brak filtra `genre`/`sort`, brak endpointu `contains-character`.

---

## Kontekst

Frontend Svelte (`svelte(wideframe)/handoff/03-api-contract.md`) oczekuje kontraktu Books który **nie odpowiada** stanowi backendu:

| Endpoint wymagany przez frontend | Stan backendu |
|---|---|
| `GET /books?q=&genre=&sort=&page=` → `Paginated<Book>` | Płaska lista, tylko `q`, brak paginacji |
| `GET /books/contains-character?name=` → `Paginated<Book>` | nie istnieje |
| `GET /books/:idOrSlug` → `Book` | tylko `:id` (int PK) |
| `GET /books/:idOrSlug/reviews?page=` → `Paginated<Review>` | review jest globalny `?book_id=`, bez paginacji |
| `POST /books/:id/reviews` → `Review` | review jest globalny `POST /api/reviews/` z `bookId` w body |

Phase 2.1 dostarcza **wszystkie pięć** endpointów Books w kształcie zgodnym z kontraktem. Vue umiera w Phase 2.6 — **brak wstecznej kompatybilności**: zmiany URL i payloadów są swobodne. Vue w międzyczasie zostaje funkcjonalny dzięki temu że jego wywołania (`GET /api/books/`, `GET /api/reviews/?book_id=`) zostają **dosłownie zachowane jako działające** w okresie przejściowym 2.1 → 2.6 (paginacja jest opcjonalna — domyślny `per_page` zwraca pełne dane jeśli pasują w jednej stronie, a Vue ignoruje meta).

> **Uwaga decyzyjna (Vue compatibility):** alternatywą jest natychmiastowe przepięcie Vue na nowy kształt — odrzucone, bo Phase 2.1 trafia do `main` zanim Phase 2.6 usunie Vue. Vue **musi działać** przez cały okres 2.1-2.5. Stąd: stare URL-e zostają jako alias, nowy kształt jest dodatkiem. Vue nie używa `contains-character`, więc tego nie trzeba aliasować.

## Obszar 1 — Slug na Book + idOrSlug routing

### Stan obecny

- `books/models.py:5-29` — `Book` ma `title`, `isbn`, brak `slug`
- `books/urls.py:7` — routing tylko po `<int:pk>`
- ISBN jest opcjonalny (`null=True`), więc nie nadaje się jako stable identifier dla URL

### Planowane zmiany

1. **Model `Book.slug`** (`books/models.py`):
   - `slug = models.SlugField(max_length=255, unique=True, blank=True, default="")`
   - Generacja w `save()`: `slugify(title)` + numeric suffix przy kolizji (utility `_generate_unique_slug(title)` w `books/models.py` lub osobny `books/slug_utils.py`)
   - `blank=True` żeby admin/seed mogły zostawić puste — wtedy `save()` wypełnia
   - **Decyzja:** slug **nie zmienia się** przy update tytułu (stable URL); jeśli admin potrzebuje refresh, kasuje pole ręcznie i save'uje
2. **Migracja danych** — dwustopniowa (jedna migracja, dwie operacje):
   - `AddField(slug, blank=True, null=True)` (tymczasowo)
   - `RunPython(generate_slugs_for_existing)` — iteruje `Book.objects.all()`, ustawia slug
   - `AlterField(slug, unique=True, blank=True, default="")` — finalizuje
3. **URL pattern `<idOrSlug>`** (`books/urls.py`):
   - `path("<str:id_or_slug>/", BookRetrieveView.as_view())`
   - `path("<str:id_or_slug>/reviews/", ...)` — patrz Obszar 4
4. **`BookRetrieveView` lookup logic** (`books/views.py:33`):
   - Override `get_object()`: jeśli `id_or_slug.isdigit()` → lookup po `pk`, inaczej po `slug`
   - Zwraca 404 dla obu ścieżek przez `get_object_or_404`
5. **Serializery — `slug` w odpowiedzi:**
   - `BookListSerializer` (`books/serializers.py:43`) + `BookDetailSerializer:65` — dodać `slug`
   - To pole pozwala Svelte budować linki bez znajomości int ID
6. **Backward-compat:** routing `<int:pk>/` zostaje jako alias **w tym samym URLconf** (zarejestrowany jako pierwszy żeby `<str:id_or_slug>` go nie złapał — Django i tak robi greedy match, więc int path wygrywa dla cyfr). Vue używa `GET /api/books/<id>/` — zachowane.

### Non-goals (Phase 2.1)

- Slug-versioning (history of slug changes z redirects) — niepotrzebne, slug stabilny
- Localized slugs (PL/EN) — projekt EN-only
- Slug edytowalny przez API — tylko Django Admin może ręcznie nadpisać

## Obszar 2 — Paginacja na listach (Books, Reviews)

### Stan obecny

- `config/settings/base.py:96-97` — global `DEFAULT_PAGINATION_CLASS = PageNumberPagination`, `PAGE_SIZE = 50`
- Wszystkie listy mają jawne `pagination_class = None` (`books/views.py:18`, `reviews/views.py:20`, `shelf/views.py:21`, `library/views.py:13/33/53`, `users/views.py:218`)
- Frontend Svelte chce shape: `{ data, page, per_page, total }` (`03-api-contract.md:12`)
- Domyślny `PageNumberPagination` zwraca `{ count, next, previous, results }` — **niezgodne**

### Planowane zmiany

1. **Custom paginator** — `config/pagination.py`:
   ```python
   from rest_framework.pagination import PageNumberPagination
   from rest_framework.response import Response

   class StandardPagination(PageNumberPagination):
       page_size = 20
       page_size_query_param = "per_page"
       max_page_size = 100

       def get_paginated_response(self, data):
           return Response({
               "data": data,
               "page": self.page.number,
               "per_page": self.get_page_size(self.request),
               "total": self.page.paginator.count,
           })
   ```
   - Klucze **snake_case** (`per_page`) — zgodne z `03-api-contract.md:12`
2. **Update `config/settings/base.py:96-97`:**
   - `DEFAULT_PAGINATION_CLASS = "config.pagination.StandardPagination"`
   - `PAGE_SIZE = 20` (kontrakt zakłada `per_page=20` jako default)
3. **`BookListView` — usunąć `pagination_class = None`** (`books/views.py:18`) — przejmie default
4. **`ReviewListCreateView` — usunąć `pagination_class = None`** (`reviews/views.py:20`) tylko dla GET — patrz Obszar 4 (review scope move)
5. **Pozostałe listy zostają płaskie** — `shelf`, `library` (authors/genres/series), `users` nie są w zakresie Phase 2.1 (osobne specs lub poza scope MVP)
6. **Backward-compat dla Vue:**
   - Vue oczekuje `GET /api/books/` → array. Nowy kształt to obiekt `{ data, ... }` — **break dla Vue**
   - **Decyzja:** dodać query param `?paginate=false` który **wymusza** stary płaski format dla **endpointów listy** (Vue używa) — implementacja: w `BookListView.list()` override sprawdza param i zwraca raw `serializer.data` jeśli `paginate=false`
   - Vue dostosować w jednym commicie (`refactor(frontend): add paginate=false to books list call`) — to jest **odstępstwo** od reguły "brak wstecznej kompatybilności" ale konieczne bo Vue żyje 5 więcej iteracji
   - Po Phase 2.6 (Vue removal) — usunąć `?paginate=false` w **Phase 2.6** w ramach cleanup

### Non-goals (Phase 2.1)

- Cursor pagination — `PageNumberPagination` wystarczy do ~10k książek
- Paginacja na wszystkich endpointach (shelf, authors, genres) — frontend ich nie potrzebuje (są używane jako selecty)
- Konfigurowalny default `page_size` przez env — hardcoded 20 zgodnie z kontraktem

## Obszar 3 — Filtry: genre, sort, q

### Stan obecny

- `books/views.py:20-30` — tylko `?q=` (icontains na title/author/genre)
- Brak osobnego filtra `genre` (po nazwie/id)
- Brak `sort` — domyślny ordering Book to brak (`Book.Meta` nie ma `ordering`)

### Planowane zmiany

1. **Filtr `?genre=` — single value** (`books/views.py:20`):
   - Akceptuje **nazwę** genre (case-insensitive exact match) — frontend ma genre jako string w UI
   - `qs = qs.filter(genres__name__iexact=genre)` + `.distinct()` (M2M)
   - Alternatywnie po `id` jeśli `genre.isdigit()` — **odrzucone**: niech będzie po nazwie, prostsze i URL-friendly
2. **Filtr `?sort=` — enum** (`books/views.py:20`):
   - `sort=rating` → `order_by("-avg_rating", "-ratings_count")`
   - `sort=recent` → `order_by("-created_at", "-id")`
   - `sort=popular` → `order_by("-ratings_count", "-avg_rating")`
   - Default (brak `sort`) → `order_by("title")` (stabilne, alfabetyczne)
   - Invalid value → ignorowane (fallback do default), brak 400
3. **Filtr `?q=` — bez zmian** (już działa na title/author/genre icontains)
4. **Walidacja query params** — przez `drf-spectacular` extension lub manual w `get_queryset()`:
   - Decyzja: **manual** — DRF nie ma natywnego filter backend dla custom logic, `django-filter` byłby overkill dla 3 paramów
   - Walidacja `sort` enum → silent fallback (nie 400), żeby nie krzyczeć przy literówkach od użytkownika
5. **Combine z paginacją** — kolejność: filtr (`q`, `genre`) → sort → paginator (DRF robi to automatycznie po `get_queryset()`)
6. **Testy** (`books/tests/test_books.py` + new `test_books_filters.py`):
   - `?genre=Fantasy` zwraca tylko książki tego gatunku
   - `?sort=rating` zwraca posortowane po `avg_rating` desc
   - `?q=austen&sort=recent` — kombinacja działa
   - `?sort=garbage` fallback do default (alfabetyczne)

### Non-goals (Phase 2.1)

- Multi-value genre (`?genre=Fantasy&genre=Sci-Fi`) — frontend ma single select
- Sort po custom polach (page_count, year) — nie ma w wireframe
- Filter po author/tag — frontend nie eksponuje (`q` wystarcza)
- Full-text search PostgreSQL (`SearchVector`) — `icontains` wystarczy do <5k książek

## Obszar 4 — Reviews scoped pod Book

### Stan obecny

- `reviews/urls.py:5-7` — routing global: `/api/reviews/` + `/api/reviews/<pk>/`
- `reviews/views.py:11` `ReviewListCreateView` — `GET ?book_id=` + `POST` z `bookId` w body (`reviews/serializers.py:28-43`)
- Frontend Svelte chce:
  - `GET /books/:idOrSlug/reviews?page=` → `Paginated<Review>`
  - `POST /books/:id/reviews` → `Review`

### Planowane zmiany

1. **Nowy endpoint `books/<id_or_slug>/reviews/`** — montowany w `books/urls.py`:
   - `path("<str:id_or_slug>/reviews/", BookReviewListCreateView.as_view())`
   - View żyje w `reviews/views.py` jako `BookReviewListCreateView` (importowany przez `books/urls.py`)
2. **`BookReviewListCreateView`** (`reviews/views.py`):
   - Inherits z `generics.ListCreateAPIView`
   - `get_book()` — helper rozwiązujący `id_or_slug` (analogicznie do Obszar 1; potencjalnie wyodrębnić do `books/lookups.py:resolve_book(id_or_slug) -> Book`)
   - `get_queryset()` → `Review.objects.filter(book=self.get_book()).select_related("user", "book")`
   - `perform_create()` → `serializer.save(user=self.request.user, book=self.get_book())`
   - `ReviewCreateSerializer` — usunąć `bookId` z `fields`/`validate` (book teraz z URL) lub stworzyć osobny `ScopedReviewCreateSerializer`
   - Permissions: GET → AllowAny, POST → IsAuthenticated
3. **Pagination** — przejmuje domyślny `StandardPagination` (Obszar 2)
4. **`POST /api/reviews/` (legacy global)** — **zostaje** jako alias dla Vue, dopóki Vue nie zniknie:
   - `ReviewListCreateView` w `reviews/views.py:11` — bez zmian
   - W Phase 2.6 (Vue removal) — usunąć ten view i `/api/reviews/` URL (oprócz `/api/reviews/<pk>/` detail, który może zostać jeśli przydatny)
5. **`ReviewDetailView` — bez zmian** (`reviews/views.py:55`): `GET/PUT/PATCH/DELETE /api/reviews/<pk>/` zostaje. Frontend Svelte używa do edit/delete (kontrakt nie określa wprost, ale CRUD na review wymaga jakiegoś detail endpointu — `/api/reviews/:pk/` jest OK, alternatywą byłoby `/books/:id/reviews/:reviewId` ale to nadmiar)
6. **Testy** (`reviews/tests/test_book_scoped.py` — nowy plik):
   - `GET /api/books/<id>/reviews/` zwraca tylko reviews tej książki, paginated
   - `GET /api/books/<slug>/reviews/` działa po slug
   - `POST /api/books/<id>/reviews/` z body `{rating, content}` (bez `bookId`) tworzy review
   - 404 dla nieistniejącej książki

### Non-goals (Phase 2.1)

- Routing `/books/:id/reviews/:reviewId/` (nested detail) — `/api/reviews/<pk>/` wystarczy
- ViewSet refactor (router-based) — generics wystarczą, mniej magii

## Obszar 5 — Endpoint `contains-character`

### Stan obecny

- Nie istnieje
- `BookCharacter` (`analysis/models.py:4-18`) — `unique_together(name, book)`, name jest **per-book** (nie globalny — ADR-003)
- Frontend chce: `GET /books/contains-character?name=Hermione` → `Paginated<Book>` — książki w których występuje postać o tej nazwie

### Planowane zmiany

1. **URL** (`books/urls.py`):
   - `path("contains-character/", BookContainsCharacterView.as_view())`
   - **Kolejność URL ma znaczenie:** ten path musi być **przed** `<str:id_or_slug>/` żeby się nie zlapał jako slug. Django próbuje paterns po kolei
2. **View** `BookContainsCharacterView` (`books/views.py`):
   - Inherits z `generics.ListAPIView`, używa `BookListSerializer` + `StandardPagination`
   - `get_queryset()`:
     ```python
     name = self.request.query_params.get("name", "").strip()
     if not name:
         return Book.objects.none()
     return (
         Book.objects.filter(characters__name__icontains=name)
         .select_related("serie")
         .prefetch_related("authors", "genres", "tags")
         .distinct()
         .order_by("title")
     )
     ```
   - Brak `name` lub pusty → zwraca pustą paginated response (nie 400 — frontend może wysłać debounce-empty)
3. **Permissions** — `AllowAny` (publiczny endpoint discovery)
4. **Testy** (`books/tests/test_contains_character.py`):
   - Książka z postacią "Hermione" w `BookCharacter` → znaleziona przy `?name=Hermione`
   - Case-insensitive match (`?name=hermione`)
   - Substring match (`?name=Herm`)
   - Brak duplikatów jeśli ta sama książka ma 2 BookCharacter pasujących (test `.distinct()`)
   - Brak `name` param → pusta paginated response (200, `data=[]`, `total=0`)

### Non-goals (Phase 2.1)

- Cross-book character search (gdyby istniał globalny `Character`) — ADR-003 mówi że nie ma; cross-book wycięte z MVP (decyzja 2026-05-23)
- Fuzzy match / trigram — `icontains` wystarczy
- Filtr po `mention_count` (minimum mentions) — frontend nie eksponuje
- Endpoint odwrotny "characters across books" — wycięte z MVP (cross-book disambiguation = Phase 2.5, alias merge **per-book only**)

## Out of scope (dla Phase 2.1)

### Decyzje grilling-sessions 2026-05-23 (explicite poza zakresem)

- **Cross-book Character** — `contains-character` szuka po `BookCharacter.name` (per-book entity z ADR-003). Brak globalnego modelu `Character`. Nie tworzymy żadnych globalnych encji postaci, nie agregujemy postaci między książkami. (To jest poza całym MVP — wraca jako Phase 3 z ADR-005.)
- **Activity feed** — żadnego endpointu `/u/:handle/activity` nie dodajemy. Frontend używa "Recently read" 6 covers (osobny endpoint w Phase 2.4 jeśli w ogóle, nie tutaj).
- **AI chat (SSE)** — `POST /books/:id/ai/ask` z `03-api-contract.md` — nie implementujemy. Łamałoby ADR-002 (text retention) + wymaga ASGI.
- **Verify/reject AI characters** — endpointy `/books/:id/ai/extraction/:characterId/verify|reject` — nie w Phase 2.1. Trafia do Phase 2.2 (AI extraction) jako minimalny `POST /books/:id/characters/:id/hide`.
- **Backward compatibility z Vue na poziomie URL/shape** — robimy jedynie minimum (`?paginate=false` flag, zachowany `POST /api/reviews/`). Vue umiera w Phase 2.6 i wtedy aliasy znikają.

### Inne wycięcia

- **Routing `/me/books/...`** (książki użytkownika) — to jest shelf, nie books; Phase 2.3 (Multi-shelf collections)
- **`PATCH /books/:id`** (edycja przez user) — books są tylko-admin (Django Admin upload)
- **`POST /books`** (user-created books) — books są tylko-admin
- **Endpoint `/books/:id/characters`** (lista postaci) — frontend Svelte używa, ale wchodzi w skład Phase 2.2 (AI extraction) — tam jest pełen kontrakt postaci + soft-delete `is_hidden` + `source`/`confidence`/`slug`
- **Endpoint `/books/:id/relations`** — jak wyżej, Phase 2.2
- **`GET /characters/:idOrSlug`, `/characters/:id/appearances`, `PATCH /characters/:id`** — wszystkie cross-book, wycięte z MVP
- **Throttling na `GET /books/`** — chroniony jest tylko auth (Phase 1)
- **Caching response** — brak (low traffic)
- **Sort po `published_year`, `page_count`** — nie ma w wireframe, dorzucimy gdy frontend poprosi
