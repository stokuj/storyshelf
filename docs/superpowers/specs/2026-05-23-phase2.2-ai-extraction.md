# Phase 2.2 — AI extraction API

**Data:** 2026-05-23
**Zakres:** `backend-django/analysis/`, `backend-django/books/` (status + ai endpointy), migracje, OpenAPI snapshot
**Stan wyjściowy:** Po Phase 2.0 (snapshot OpenAPI, CORS+cookies cross-origin) i Phase 2.1 (Books API: paginacja, `slug` na `Book`, `contains-character`). Pipeline NER+LLM działa, triggerowany tylko z Django Admin akcją `analyze_selected_books` (`analysis/admin.py:40-48`). Brak HTTP endpointów do triggera/odczytu statusu ekstrakcji. `BookCharacter` ma tylko `name`/`description`/`mention_count` (`analysis/models.py:4-18`). `analyse_book` jest semi-idempotentny (deletes + creates wszystkie encje, `analysis/tasks.py:44-57`) — gubi historię `is_hidden` przy re-runie i nie wystawia statusu.

---

## Kontekst

Frontend Svelte (Phase 2.7-2.9) potrzebuje:

1. **HTTP triggera ekstrakcji AI** zamiast wejścia do Django Admin — admin uruchamia analizę przyciskiem w UI książki.
2. **Statusu ekstrakcji** (`pending|running|done|failed`) na `Book` żeby UI mógł pokazać badge / spinner / błąd bez pollingu Flower / kolejek Celery.
3. **Stabilnego `slug`** na `BookCharacter` — routing `/books/:id/characters/:slug` (handoff `04-routes-and-screens.md`, prompt `07-character-pages.md`). `id` z auto-inkrementu zmienia się po re-runie, slug po `name` jest stabilny.
4. **Atrybutów `source` + `confidence`** na `BookCharacter` — handoff `02-data-model.md:65-66` wymaga ich w typie `Character`. `source` rozróżnia `human|ai|ai-verified`, `confidence` to `0..1` (tylko dla AI).
5. **Soft-delete `is_hidden`** — admin ukrywa fałszywie pozytywne charaktery (np. miejsca błędnie sklasyfikowane przez NER). Hard delete tracimy przy re-runie; soft-delete + idempotentny upsert zachowuje decyzję admina pomiędzy runami.
6. **Idempotentnego `analyse_book`** — re-run NIE usuwa charakterów z `is_hidden=True` i NIE resetuje `source`/`confidence` przy istniejących encjach (update mention_count, zachowaj resztę).

Phase 2.2 zamyka backend pod prompt `06-ai-cast-panel.md` (AICastPanel) i `07-character-pages.md` (per-character routing). Wszystkie endpointy AI są **admin-gated** (`IsAdminUser`) — patrz decyzje grilling.

## Obszar 1 — Pola na `BookCharacter` (`slug`, `source`, `confidence`, `is_hidden`)

### Stan obecny

- `analysis/models.py:4-18` — `BookCharacter` ma tylko `book`/`name`/`description`/`mention_count`.
- `BookCharacterSerializer` (`books/serializers.py:8-13`) zwraca `id`, `name`, `description`, `mentionCount`.
- `analyse_book` (`analysis/tasks.py:44-57`) robi delete + create — gubi wszystko z historii.

### Planowane zmiany

1. **Migracja modelu** (`analysis/migrations/0004_bookcharacter_ai_fields.py`):
   - `slug` — `SlugField(max_length=255)`, generowany z `name` (`django.utils.text.slugify`), unique per book.
     - `UniqueConstraint(fields=["slug", "book"], name="unique_character_slug_per_book")`.
     - Backfill w data migration: dla istniejących rekordów wygenerować `slug = slugify(name)`; przy kolizji w obrębie tej samej książki dopisać sufix `-2`, `-3` (deterministyczny).
   - `source` — `CharField(max_length=20, choices=Source.choices, default=Source.AI)`. Enum `Source`: `HUMAN="human"`, `AI="ai"`, `AI_VERIFIED="ai-verified"`. Backfill: `AI` dla istniejących (wszystkie powstały z pipeline'u).
   - `confidence` — `FloatField(null=True, blank=True)`. Validators: `MinValueValidator(0.0)`, `MaxValueValidator(1.0)`. Backfill: `None`.
   - `is_hidden` — `BooleanField(default=False)`. Backfill: `False`.
2. **`BookCharacter.save()` override** — jeśli `slug` pusty, wygeneruj z `name` przed save (dla create przez ORM bez explicit slug). Logika deterministyczna: kolizja → sufix numeryczny.
3. **`analysis/admin.py`** — `BookCharacterAdmin.list_display` dodać `slug`, `source`, `is_hidden`. `list_filter` dodać `source`, `is_hidden`. Akcja masowa `mark_hidden` / `mark_visible`.
4. **`books/serializers.py:8-13` — `BookCharacterSerializer`**:
   - Dodać pola: `slug`, `source`, `confidence`, `isHidden`.
   - W `BookDetailSerializer.to_representation` (`books/serializers.py:107-110`) — filtrować `instance.characters.filter(is_hidden=False)` dla anonimowych / non-admin. Admin widzi wszystkie (`is_hidden=True` z badge w UI).
5. **`books/views.py:42-49` — `BookRetrieveView`**:
   - `prefetch_related("characters")` zostaje, ale dla non-admin użyć `Prefetch("characters", queryset=BookCharacter.objects.filter(is_hidden=False))`.

### Non-goals (Phase 2.2)

- Cross-book `Character` (globalna postać z `book_ids` jak w handoff `02-data-model.md:67-68`) — encje pozostają per-book (ADR-003). Pole `book_ids` w typie TS na froncie wypełniamy `[book.id]` jako workaround.
- `also_known_as` (aliasy) — wchodzi w Phase 2.5 (character disambiguation / alias merge w obrębie książki).
- `role` / `archetype` / `short_bio` / `long_bio` / `tags` z `02-data-model.md:54-69` — brak na backendzie i nie wchodzi w MVP (LLM nie generuje, manualne pola admina są out of scope).

## Obszar 2 — `ai_extraction_status` na `Book`

### Stan obecny

- `books/models.py:5-29` — brak pola statusu. `BookDetailSerializer.to_representation` (`books/serializers.py:91-93`) zwraca `analysisStatus.analysisFinished` jako `instance.characters.exists()` — sygnał binarny, nie odróżnia pending/running/failed.
- `analyse_book` task (`analysis/tasks.py:21-65`) nie zapisuje stanu nigdzie; jeśli failuje, `Book.text` zostaje (delete pod koniec) i admin nie wie czy run się wywalił.

### Planowane zmiany

1. **Migracja** (`books/migrations/000X_book_ai_extraction_status.py`):
   - Pole `ai_extraction_status` — `CharField(max_length=20, choices=AIExtractionStatus.choices, default=NONE)`.
   - Enum w `books/models.py`:
     ```python
     class AIExtractionStatus(models.TextChoices):
         NONE = "none"          # nigdy nie odpalone
         PENDING = "pending"    # zakolejkowane / zaraz startuje
         RUNNING = "running"    # task wystartował (mid-run)
         DONE = "done"          # zakończone sukcesem
         FAILED = "failed"      # task wyleciał lub LLM odmówił
     ```
   - Pole `ai_extraction_started_at` — `DateTimeField(null=True, blank=True)`.
   - Pole `ai_extraction_finished_at` — `DateTimeField(null=True, blank=True)`.
   - Pole `ai_extraction_failure_reason` — `TextField(blank=True, default="")`. Krótki komunikat błędu (`<512` znaków, truncate w kodzie).
   - Backfill: dla książek z `characters.exists()` → `DONE`; reszta → `NONE`.
2. **`analyse_book` (`analysis/tasks.py:21-65`)** — ustawia status:
   - Przed startem: `Book.objects.filter(id=book_id).update(ai_extraction_status="running", ai_extraction_started_at=timezone.now(), ai_extraction_failure_reason="")`.
   - Po sukcesie (po `relations_for_book.delay`): `update(ai_extraction_status="done", ai_extraction_finished_at=timezone.now())`.
   - W `try/except` wokół całego NER: na exception `update(ai_extraction_status="failed", ai_extraction_finished_at=timezone.now(), ai_extraction_failure_reason=str(e)[:512])` + `raise` (żeby Celery zalogowało).
3. **`relations_for_book` (`analysis/tasks.py:68-110`)** — NIE zmienia statusu książki (status `done` ustawiany w `analyse_book` po dispatch'u LLM, bo LLM jest fire-and-forget i błędy LLM są per-para, nie blokują całości — patrz `analysis/tasks.py:91`).

### Non-goals (Phase 2.2)

- Statusowanie LLM per-relacja (`CharacterRelationship.source`/`confidence`) — wchodzi w Phase 4 (AI/LLM pipeline expansion). MVP: relacje są jednorodne (`source="ai"`, brak confidence), bez UI verify.
- Webhooks / SSE notyfikacje o zmianie statusu — Svelte poll'uje `GET /books/:id/ai/extraction` z UI (interval 3s gdy `running`, stop po `done|failed`).
- Per-task progress (`covered_through.percentage` z `02-data-model.md:150`) — brak instrumentacji chunkowania. W odpowiedzi zwracamy stub `{ chapter: null, percentage: status === "done" ? 100 : null }`.

## Obszar 3 — Endpointy HTTP `/books/:id/ai/extract` i `/books/:id/ai/extraction`

### Stan obecny

- Brak endpointów w `books/urls.py:1-9`. Trigger tylko przez Django Admin (`analysis/admin.py:40-48`).
- Brak app `analysis.urls` / `analysis.views`.

### Planowane zmiany

1. **Nowy plik `analysis/views.py`** — DRF `APIView` z dwoma endpointami:
   - `POST /api/books/<int:book_id>/ai/extract/` — `AIExtractionTriggerView`:
     - Permissions: `IsAdminUser` (hardcoded, **nie** subclass — patrz decyzje grilling).
     - Walidacja: `Book.objects.get(id=book_id)` → 404 jeśli brak.
     - **Idempotentny od strony HTTP:** jeśli `book.ai_extraction_status in ("pending", "running")` → 200 z cached `AIExtraction` (no-op, nie dispatchujemy ponownie). To uniemożliwia race / DDoS na celery-ner.
     - Jeśli `book.text` puste i `status == "done"` → 200 z cached (nic do roboty, encje są).
     - W innym wypadku: `Book.objects.filter(id=book_id).update(ai_extraction_status="pending", ai_extraction_failure_reason="")`, `analyse_book.delay(book_id)`, 202 Accepted z `AIExtraction` body.
   - `GET /api/books/<int:book_id>/ai/extraction/` — `AIExtractionRetrieveView`:
     - Permissions: `IsAdminUser`.
     - 404 jeśli book nie istnieje.
     - Zwraca `AIExtraction` payload (zob. niżej).
2. **`analysis/serializers.py`** (nowy plik) — `AIExtractionSerializer`:
   ```python
   {
     "id": str(book.id),                   # frontend traktuje extraction jako 1-1 z book
     "book_id": book.id,
     "status": book.ai_extraction_status,  # none|pending|running|done|failed
     "created_at": book.ai_extraction_started_at,  # ISO8601 lub null
     "finished_at": book.ai_extraction_finished_at,
     "characters": BookCharacterSerializer(book.characters.all(), many=True).data,  # admin widzi wszystkie (z is_hidden)
     "relations": CharacterRelationSerializer(book.character_relationships.all(), many=True).data,
     "covered_through": {"chapter": None, "percentage": 100 if status == "done" else None},
     "confidence_summary": {
        "overall": <avg confidence z nie-hidden charakterów AI, lub None>,
        "flagged_low": <count z confidence < 0.5>,
     },
     "failure_reason": book.ai_extraction_failure_reason or None,
   }
   ```
3. **`analysis/urls.py`** (nowy plik):
   ```python
   urlpatterns = [
       path("books/<int:book_id>/ai/extract/", AIExtractionTriggerView.as_view()),
       path("books/<int:book_id>/ai/extraction/", AIExtractionRetrieveView.as_view()),
       path("books/<int:book_id>/characters/<int:character_id>/hide/", BookCharacterHideView.as_view()),
   ]
   ```
4. **`config/urls.py:5-21`** — dodać `path("api/", include("analysis.urls"))`. UWAGA: route'y są zagnieżdżone w `/books/` ale serwowane z app `analysis` (semantyka domeny). Alternatywa: trzymać w `books/urls.py` — odrzucam, bo `analysis/` jest właścicielem ekstrakcji.

### Non-goals (Phase 2.2)

- **`POST /books/:id/ai/extract/refresh`** (force re-run) — **WYCIĘTE z MVP**. Admin może w Django Admin użyć tej samej akcji `analyze_selected_books`. Wymagałoby decyzji jak interakuje z `is_hidden` — odkładamy do Phase 4.
- **`POST /books/:id/ai/extraction/:characterId/verify`** — **WYCIĘTE z MVP** (grilling: verify/reject UI poza zakresem MVP).
- **`POST /books/:id/ai/extraction/:relationId/verify`** — **WYCIĘTE z MVP**.
- **`POST /books/:id/ai/ask`** (SSE chat) — **WYCIĘTE z MVP** (grilling: złamałoby ADR-002 text retention + wymaga ASGI).
- Otwarcie endpointów `IsAdminUser` → autoryzowanym userom — decyzja na Fazę 3 z rate-limitem.

## Obszar 4 — Soft-delete `POST /books/:id/characters/:characterId/hide`

### Stan obecny

- Brak endpointu. Admin musi usuwać charakter ręcznie w Django Admin, co przy następnym `analyse_book` wraca (delete + recreate w `analysis/tasks.py:45-50`).

### Planowane zmiany

1. **`analysis/views.py` — `BookCharacterHideView`**:
   - `POST /api/books/<int:book_id>/characters/<int:character_id>/hide/`.
   - Permissions: `IsAdminUser`.
   - Body: opcjonalny `{"hidden": true|false}` (default `true`). Pozwala odhide'ować (admin może cofnąć).
   - Walidacja: `BookCharacter.objects.get(id=character_id, book_id=book_id)` → 404.
   - `character.is_hidden = bool(request.data.get("hidden", True)); character.save(update_fields=["is_hidden"])`.
   - Response: zaktualizowany `BookCharacterSerializer` (200).
2. **NIE jest to verify/reject UI** — zgodnie z grilling-decyzją. Po prostu flag boolean w jedną/drugą stronę. UI Svelte używa tego do "ukryj ten charakter z listy" w panelu admina.

### Non-goals (Phase 2.2)

- Audit log kto kiedy ukrył — Phase 4 (admin actions logging).
- Reason / komentarz przy hide — Phase 4.
- Endpoint po `slug` zamiast `id` — `id` wystarczy bo admin ma w UI listę z DB.

## Obszar 5 — Idempotentność `analyse_book`

### Stan obecny

- `analysis/tasks.py:44-57` — `delete()` na wszystkich `BookCharacter`/`BookPlace`/`BookOrganization` przed `create()`. Konsekwencje:
  - `is_hidden=True` ginie przy re-runie.
  - `source` / `confidence` zresetowane (gdyby admin ręcznie ustawił `AI_VERIFIED`).
  - `id` charakterów się zmienia — psuje URL'e `/books/:id/characters/:id/`. Slug temu zaradza, ale i tak nie chcemy gubić rekordów.
  - `CharacterRelationship` z `CASCADE` na FK do `BookCharacter` znika razem z charakterami.

### Planowane zmiany

1. **Refactor `analyse_book` (`analysis/tasks.py:21-65`) — upsert zamiast delete+create**:
   - Dla `BookCharacter`:
     - `existing = {c.slug: c for c in BookCharacter.objects.filter(book=book)}`.
     - Dla każdej nowej encji z NER (`name`, `count`):
       - `slug = slugify(name)` (z deduplication w obrębie batcha — kolizja → sufix).
       - Jeśli `slug in existing` → `update(mention_count=count)`, zachowaj `is_hidden`, `source`, `confidence`, `description`.
       - Jeśli `slug not in existing` → `create(slug=slug, name=name, mention_count=count, source="ai", confidence=None, is_hidden=False)`.
     - Encje które były w DB ale NER ich tym razem nie znalazł → **pozostają** (nie hard-deletujemy; mention_count zostaje stary). Decyzja: nie chcemy stracić ręcznie zweryfikowanych encji jeśli NER z chunkowanym tekstem ich tym razem nie podchwycił.
       - **Alternatywa odrzucona:** `BookCharacter.objects.filter(book=book, source="ai").exclude(slug__in=new_slugs).delete()` — niebezpieczne, bo NER jest stochastyczny.
   - Dla `BookPlace`/`BookOrganization` — analogicznie (slug + upsert, choć tam nie ma `source`/`confidence`/`is_hidden` w MVP).
2. **`CharacterRelationship`** — `relations_for_book` używa `get_or_create` (`analysis/tasks.py:102-107`) już teraz, ale jeśli zostawiamy stare encje, stare relacje też zostają. Decyzja: relacje też zostawiamy (idempotentne `get_or_create` per (from, to, book)).
3. **Logowanie**: log INFO przy każdym `upsert` z licznikami `created`/`updated`/`untouched` — pomaga admina ocenić jakość re-runu.

### Non-goals (Phase 2.2)

- "Diff mode" w UI (pokaż co dodano/zmieniono w re-runie) — Phase 4.
- Versioning `AIExtraction` (historia runów) — Phase 4. Obecnie 1 book = 1 ekstrakcja (aktualny stan).
